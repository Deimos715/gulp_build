# Project Style

Переносимые правила верстки и оформления кода. Файл можно класть в корень любого проекта и использовать как локальный стандарт для новых задач.

## Общий подход

### Для нового проекта

- До начала разработки определить структуру файлов, правила именования, методологию классов и точки входа.
- Выбранные соглашения применять последовательно во всем проекте.
- Не добавлять новые библиотеки и абстракции без явной необходимости.
- Верстать от дизайн-системы и макета, формируя понятные и переиспользуемые проектные паттерны.
- Если макет конфликтует с принятыми правилами, зафиксировать конфликт и выбрать решение осознанно.

### Для существующего проекта

- Учитывать уже согласованные реализации, структуру и стиль проекта.
- Не переписывать блок целиком, если задача решается точечной правкой.
- Не добавлять новые библиотеки и абстракции без явной необходимости.
- Не менять структуру файлов, правила именования и форматирование без причины.
- Верстать от дизайн-системы и макета, учитывая существующие проектные паттерны.
- Если макет конфликтует с проектными правилами, зафиксировать конфликт и выбрать решение осознанно.

## HTML

- Использовать семантическую структуру: `header`, `main`, `section`, `nav`, `footer`, `article`, `button`, `form`.
- Крупный смысловой экран оформлять отдельной секцией.
- Внутри секции использовать контейнер проекта: `.container` или локальный аналог, если он уже принят.
- Не плодить лишние обертки. Обертка нужна только для сетки, позиционирования, состояния или переиспользования.
- Повторяемые части выносить в компоненты/partials/includes, если такая структура уже есть в проекте.
- Текстовые заголовки должны идти по иерархии: один `h1` на страницу, далее `h2`, `h3` и ниже по смыслу.
- Интерактивные элементы делать через `button` или `a` по назначению, а не через `div`.
- Содержательные изображения должны иметь осмысленный `alt`, а декоративные — пустой `alt=""`.

## БЭМ

- Основной подход к классам — БЭМ.
- Блок описывает самостоятельную сущность: `.header`, `.hero`, `.services`, `.case-card`, `.form`.
- Элемент пишется через двойное подчеркивание: `.hero__title`, `.hero__content`, `.services__item`.
- В новом проекте модификатор писать через двойной дефис: `.button--primary`, `.card--featured`.
- Состояния обозначать отдельными классами: `.is-active`, `.is-open`, `.is-disabled`.
- В существующем проекте сохранять уже принятый синтаксис модификаторов и состояний.
- Не делать длинные цепочки элементов: плохо `.block__item__title`; лучше отдельный блок `.card` и элемент `.card__title`.
- Не завязывать CSS на теги внутри сложных компонентов, если можно дать элементу класс.
- Не использовать глобальные классы для уникального поведения блока.

Пример:

```html
<section class="hero-section">
    <div class="container hero-section__container">
        <div class="hero-section__content">
            <p class="hero-section__eyebrow">Сложные веб-продукты для бизнеса</p>
            <h1 class="hero-section__title">Проектируем и разрабатываем цифровые продукты</h1>
            <p class="hero-section__text">Создаем веб-сервисы и личные кабинеты с продуманной архитектурой.</p>
        </div>
    </div>
</section>
```

## SCSS

- Каждый крупный блок — отдельный SCSS-файл: `_header.scss`, `_hero.scss`, `_services.scss`, `_footer.scss`.
- Общие переменные хранить в `_variables.scss`.
- Общие миксины хранить в `_mixins.scss`, если они реально переиспользуются.
- Общую базу хранить в `_global.scss`: body, container, типографика, базовые ссылки, кнопки, helpers.
- Все partials подключать через главный файл стилей проекта, например `style.scss`.
- В новом проекте использовать `@use` и `@forward` вместо устаревшего `@import`.
- В существующем проекте с `@import` не смешивать подходы без отдельной миграции.
- Внутри SCSS писать блоком и вкладывать элементы через `&__element`.
- Не вкладывать селекторы глубже 3 уровней без необходимости.
- Не писать стили по `id`, кроме технических случаев, которые уже приняты в проекте.
- Не править скомпилированный/minified CSS, если есть исходный SCSS.

Пример структуры:

```scss
@use "variables" as *;
@use "global" as *;
@use "header" as *;
@use "hero" as *;
@use "services" as *;
@use "footer" as *;
@use "adaptive" as *;
```

Пример блока:

```scss
@use "variables" as *;

.hero-section {
    position: relative;
    overflow: hidden;
    padding: 96px 0 120px;

    &__container {
        display: grid;
        grid-template-columns: minmax(0, 1fr) minmax(320px, 480px);
        gap: 64px;
        align-items: center;
    }

    &__eyebrow {
        margin: 0 0 16px;
        font-size: 12px;
        line-height: 1.2;
        font-weight: 600;
        text-transform: uppercase;
    }

    &__title {
        margin: 0;
        font-size: 64px;
        line-height: 1.08;
        font-weight: 700;
    }

    &__text {
        margin: 24px 0 0;
        font-size: 18px;
        line-height: 1.55;
    }
}
```

## Адаптив

- Использовать брейкпоинты проекта. Если их нет, выбирать точки по моменту, когда композиция перестает работать.
- В качестве стартовых ориентиров, а не привязки к конкретным устройствам, можно использовать: `1199px`, `991px`, `767px`, `575px` и `425px`.
- Адаптивные правки можно хранить в `_adaptive.scss`, если так принято в проекте.
- Если проект пишет адаптив рядом с блоком, сохранять этот подход.
- На мобильных не уменьшать все пропорционально. Перестраивать композицию: колонки становятся вертикальными, карточки идут в один столбец, декоративные элементы упрощаются.
- Текст не должен обрезаться, налезать на соседние элементы или зависеть от фиксированной высоты.
- Кнопки и интерактивные элементы на мобильных должны иметь удобную область клика.

## Сетка И Отступы

- Сетка страницы задается на уровне основного контейнера или главного фрейма макета.
- Не создавать локальные сетки внутри каждого блока, если общая сетка уже есть.
- Межколонник сетки не равен обязательному gap между всеми UI-элементами.
- Для крупных групп использовать отступы сетки и section spacing.
- Для компактных связанных контролов использовать меньший gap, если это не ломает привязку к сетке.
- Правый и левый край ключевых групп должны совпадать с контейнером или колонками.
- Не выравнивать элементы «на глаз», если можно посчитать координату от контейнера.

## Компоненты

- Кнопки, карточки, формы, бейджи и навигацию делать переиспользуемыми классами.
- Состояния `hover`, `active`, `focus`, `disabled` описывать сразу, если компонент интерактивный.
- Понятное состояние `focus-visible` обязательно для всех интерактивных элементов.
- Не отключать outline без замены на понятное состояние фокуса.
- Не делать карточку там, где контент можно показать без нее. Карточка нужна для структурирования смысла, а не как декор.
- Визуальный акцент строить через типографику, композицию, воздух и один сильный объект, а не через большое количество карточек.

## Формы

- Поля формы должны иметь `label` или доступное имя.
- Ошибки показывать рядом с полем.
- Состояния поля: default, hover, focus, error, disabled.
- Не снижать opacity всей формы для disabled, если от этого ухудшается читаемость.
- Маски, валидацию и скрипты писать в отдельном JS partial, если проект так организован.

## JavaScript

- Использовать единую точку входа JS, а логику самостоятельных блоков выносить в модули, если это оправдано структурой проекта.
- Классы состояний переключать явно: `is-open`, `is-active` или проектный аналог.
- Не смешивать логику и стили через inline-обработчики, если проект уже использует модульный JS.
- Перед добавлением новой зависимости проверить, нельзя ли решить задачу существующим кодом.

## Figma И Макет

- Перед версткой сверить макет, дизайн-систему и существующий код.
- Если в главном фрейме уже есть сетка, не создавать сетки во внутренних фреймах.
- При переносе из Figma считать реальные размеры: контейнер, колонки, gutter, отступы, высоты элементов.
- Если элемент визуально не совпадает с сеткой, проверить, является ли это ошибкой макета или осознанным исключением, и только после этого выбирать выравнивание.
- Декоративные glow/blur-объекты не должны ухудшать читаемость текста и не должны вылезать за границы секции, если это не предусмотрено макетом.
- Логотипы и брендовые элементы не перерисовывать без отдельной задачи.

## Доступность

- Контраст текста должен быть достаточным для чтения.
- Интерактивные элементы доступны с клавиатуры.
- У содержательных изображений есть осмысленный `alt`, у декоративных — `alt=""`.
- Элементы управления без видимого текста имеют доступное имя через `aria-label`, `aria-labelledby` или скрытый текст; декоративные иконки скрыты от скринридера.
- Не использовать только цвет для передачи важного состояния.

## Проверка Перед Сдачей

- Проверить страницу на desktop, tablet и mobile.
- Проверить hover/focus/active состояния.
- Проверить, что текст не обрезается и не налезает на соседние элементы.
- Проверить, что SCSS подключен через главный style-файл.
- Проверить, что новые файлы названы по блоку и не дублируют существующие partials.
- Запустить сборку проекта, если она доступна.
- Не оставлять временные классы, тестовый текст, отключенные debug-стили и мусорные изображения.
